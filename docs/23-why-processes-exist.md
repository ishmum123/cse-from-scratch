# Walls Between Programs

## The Problem

Early computers ran one program at a time. When time-sharing arrived and multiple programs coexisted, they shared the same memory. Program A stores a value at memory address 0x1000. Program B also uses address 0x1000. They overwrite each other's data. Program A calls `free(ptr)` — but `ptr` was memory that Program B was still using. B crashes with garbage data it didn't write.

Worse: Program A misbehaves — writes past the end of its array, corrupts address 0x0000 (often where the OS kernel lives), and the entire machine crashes. All other programs die with it. One buggy student's homework submission on a university server takes down everyone's sessions.

Any solution must:

- Give each program the illusion of exclusive memory ownership
- Prevent any program from reading or writing another program's memory
- Contain crashes — one program's failure must not affect others
- Work with programs that weren't written to cooperate

## Failed Attempts

### Attempt 1: Compile Programs to Use Different Address Ranges

Agree in advance: Program A uses addresses 0–1MB, Program B uses 1MB–2MB, Program C uses 2MB–3MB. Recompile each program with the right base address.

This works as long as you know at compile time which slot each program gets, and programs never exceed their allocation, and you never run more programs than you planned for. Violate any condition and you're back to collisions. The real killer: programs use pointers internally. If Program A is recompiled for address 0x100000 but the pointer `0x1004a2` is hardcoded deep in a library, moving the program to a different slot requires relinking everything. A program that uses shared libraries can't know where those libraries will be loaded. This is "static relocation" — DOS-era computing — and it scales to maybe 4 programs if you're lucky.

### Attempt 2: Bounds-Check Every Memory Access in Software

Leave memory shared but add a bounds checker: each program declares its address range, and every memory access is validated against those bounds at runtime. Out-of-bounds access is caught and the offending program is killed.

The overhead is catastrophic — every memory instruction now involves a range check, potentially 2–5× slowdown. Modern programs make millions of memory accesses per second. But the deeper problem: this relies on programs declaring their bounds honestly. A malicious or buggy program can declare "I use all of memory" and the checker approves it. Shared memory means kernel memory is also accessible — a program can overwrite its own bounds-check records or the kernel's memory maps. Software enforcement of hardware access is self-undermining.

### Attempt 3: Base and Limit Registers

Hardware provides two registers: `base` (start of this program's physical memory) and `limit` (size). The OS sets these on each context switch. Every memory address generated by the program is automatically added to `base` by hardware before hitting RAM. Any address ≥ `limit` causes a hardware fault.

This is a genuine improvement — hardware enforcement, no software overhead, automatic relocation. But it's one contiguous region per program. Programs can't have separate code, heap, and stack regions that grow independently. You can't share read-only code between two copies of the same program. Swapping a program to disk requires saving and restoring its entire region. And you still can't run a program that needs more memory than your physical RAM.

## The Discovery

Static relocation fails at scale. Software bounds-checking is too slow and self-undermining. Base/limit registers work but leave programs as flat, indivisible blobs.

The real solution separates two things that had always been conflated: **addresses a program uses** and **physical memory locations**. Give each program its own private address space — its addresses run from 0 to some large number — and have hardware translate those addresses to physical locations transparently. Program A's address 0x1000 and Program B's address 0x1000 map to *different* physical locations. The translation table lives in protected memory only the OS can modify.

This is the **process** abstraction. A process is not just a running program — it's a program plus a private address space, plus private file descriptor table, plus private signal handlers, plus all the other resources that must be isolated to make independent execution safe. The OS kernel maintains one translation table (page table) per process and switches tables on every context switch.

Crash containment follows automatically: when a process crashes or is killed, the OS reclaims its memory, closes its file descriptors, and removes its page table. No other process's page table pointed to those pages. They're simply gone.

The sim tracks two service arrays: `monoState` (a single monolithic process where all components share one `{alive}` flag) and `isoState` (isolated processes, each with its own flag); `crashRandom()` kills all of `monoState` at once but only one entry in `isoState`, showing fault isolation in one click.

## When It Breaks

**Process creation overhead.** A new process requires cloning the parent's entire page table, file descriptor table, and kernel state. On Linux, `fork()` uses copy-on-write to defer physical page copies until a write occurs — but even so, `fork()` of a 10GB process copies a huge page table. High-throughput servers that fork per request (like old Apache prefork) hit this ceiling. This is why threads (chapter 24) exist: they share an address space and cost far less to create.

**Cross-process communication requires explicit channels.** Since processes can't read each other's memory by design, sharing data requires pipes, sockets, shared memory segments, or message queues — all of which require OS involvement. Two processes can't pass a pointer between them; the same address means different physical memory. This isolation that makes processes safe also makes them awkward to compose, which is why microservice architectures spend so much engineering on serialization protocols.

## Transfer

- **Browser tabs as processes.** Chrome runs each tab in a separate process (since 2008). One tab's JavaScript crash doesn't bring down others. A tab processing a malicious web page can't read another tab's cookies. The cost: each tab's process isolation uses ~50MB of overhead — observable on machines with many open tabs.
- **`fork()` + `exec()` in Unix shells.** A shell running `ls | grep foo` forks twice, creating two child processes that inherit the parent's file descriptors, then both call `exec()` to replace themselves with `ls` and `grep`. The pipe is set up before exec — the processes' isolation is the feature that makes them composable as tools.
- **Containers (Docker) as process groups.** Linux cgroups and namespaces isolate groups of processes so they see separate filesystem roots, PIDs, and network stacks — a "process" at the container level. The isolation is weaker than a VM but the overhead is far lower.
